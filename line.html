專案起源・客服人員負荷過載

身為 PHILIPS 3C 和家電產品線代理商，面對促銷檔期與新品上市帶來的流量高峰，常態性面臨客服人力缺口，導致回應遲緩、客戶滿意度下滑。

我經由數據歸納，識別出「高重複性查詢」與「缺乏客服分流機制」為根本痛點。因此我提議導入自動化客服中樞，目標在於不增加額外人力成本下，解決回應即時性的瓶頸。

專案亮點

主導執行 LINE 客服機器人的開發與 POC 驗證，成功優化客服回應效能。

規劃自動化架構

運用 n8n 作為核心，串接 LINE Webhook 與 Supabase 標籤資料庫。建立一套具備可擴充性的自動分流系統，將繁複的人工作業轉化為標準化作業流程。

標準化語意標籤資料庫

深入分析歷史客服語料，將非結構化問題透過『關鍵字』方式重構為結構化知識庫。確保服務內容的一致性，大幅降低內部 FAQ 更新的營運維護成本。

低成本 + 高效率 MVP 策略

盤點現有資源與技術，並評估地端 LLM 和雲端 LLM 成本後，選擇無伺服器架構進行 POC 驗證，成功建立 MVP，並且隨時可依未來業務量進行付費升級。

即時回應優化

透過雲端無伺服器架構，排除地端硬體瓶頸，實現 24 小時不間斷的即時回應。

解決方案流程・n8n + Supabase

透過 LINE Webhook 接收訊息，由 n8n Workflow 判斷關鍵字後，至 Supabase FAQ 資料庫比對標籤，找到對應答案再以 LINE Reply 回覆。整體達成自動化處理、精準回覆、安全可靠與彈性擴充。

專案挑戰・標籤關鍵字定義

此專案初期考慮到成本預算問題，因此以「零成本」的方式開發 MVP 進行市場 POC 驗證，依據指標成果再考慮是否投入付費成本。因專案未串接 LLM，需要針對每個問題明確定義相關的 keyword 供機器人判斷；初期內部測試發現相關詞的數量遠超預期，因此必須先列出各種問題情境，再依據『動作／目的』、『場合』、『行為約束』與『進階標籤』等分類方式整理。

標籤化意圖識別與分流設計

1

標籤化意圖識別

在訪談和歷史客服資料中發現，70% 的 LINE@ 問題是屬於可標準化回覆的『重複性需求』。我將雜亂的客戶問題收斂為『產品、訂單、保固』等核心類別，再將熱門關鍵字標籤化，建立一套「關鍵字問答表」。

2

問題分流的情境設計

在 n8n 中構建多分支 Switch 架構。基於 MVP 資源限制，我設計『價值分流策略』，將高頻、低複雜度的詢問留給機器人自動化處理；針對需個案判斷的高價值／高風險問題（例：退貨），則透過分流轉介至官網訂單留言或真人客服，確保客服人力成本配置的最優化。

3

系統架構規劃

在問題分流串接的架構中，需要考慮到『邊緣情境』，例如長問句或罕見關鍵字。因此特別設計『全域防呆引導』，確保系統在任何情況下皆能承接客戶訊息，避免已讀不回帶來的信任流失。

成效拆解・問卷與分流數據

1

問卷收集方式與樣本數

在 LINE 圖文選單處新增問卷回饋連結，提供「填完問卷送折價券」的互惠活動吸引用戶協助填寫，最後獲得 38 位用戶填寫，得到 82% 的滿意度（4.1 分，滿分 5 分）。

2

客服分流成效

透過客服機器人，成功過濾 65.7% 的瑣碎詢問，客服團隊僅需處理剩餘 34.3% 的複雜個案。

自動化分流率 =（總進線量 − 真人轉接後的案件量）÷ 總進線量
  • 問卷顯示：38 筆對話中，約有 13 筆選「是」（需要轉接真人）
  • 轉換率：(38 − 13) ÷ 38 = 65.7%
3

高風險與個資類意圖的分流轉向策略

分析轉接數據後發現，約 60.9% 的用戶仍傾向轉人工查詢訂單，主因是該類型問題涉及『個資安全』與『出貨時間敏感度』（理由為需要出國、出差時使用，因此對收貨日期壓力較高）。

未來想優化的方向

AI Agent + RAG 向量資料庫
數據觀察到，有 34.8% 的客戶反應機器人無法理解問題。下一步我規劃將產品結合 LLM（例：Gemini API），並將產品說明書 PDF 進行向量化，使用 Dify 架設 RAG 系統串接到 n8n，讓客服機器人能更清楚判斷客戶對於商品細節的問題，且大幅降低 LLM 產生幻覺的風險，讓產品具備『從規則自動化進化到 AI 自動化』。
